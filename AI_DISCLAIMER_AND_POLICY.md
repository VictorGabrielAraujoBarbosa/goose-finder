# AI In Goose Finder:

**Goose Finder** was born in spontaneity, from a random thought in a boring afternoon - "What if we made a code analysis tool inspired by untitled goose game?". To manage to capture and experiment with the nature behind that fleeting idea, we used LLMs to do rapid prototyping and come up with an MVP.

Admittedly, said MVP was a bunch of AI mumbo jumbo, but it did the thing we wanted it to do, or at least something that resembled it. It allowed us to validate our gut feeling that this would be an interesting tool, both in the utility and aesthetics sense.

Having achieved and validated that MVP (which you can see at the start of our git history), we changed our usage and policy on AI usage. We want goose finder to be a solid, reliable piece of software going forward, which just isn't something that is achievable through vibe coding. With this in mind, we decided that, **to contribute to this project using AI, you must understand the underlying principles behind your code**. It is not sufficient for the code to run and pass tests - you must review it to know how it runs, what it does, which decisions were made and the reason behind those choices.

This is the standard that we follow, and expect you to follow as well. Yes, you can use AI completion, code generation, and other utility features no problem. But, at the end of the day, you must be the mind behind those changes - **the LLM is your assistant, not the other way around.**
